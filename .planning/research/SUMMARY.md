# Project Research Summary

**Project:** the shadow realm
**Domain:** Personal landing page / project catalog hub (static Next.js site on Vercel)
**Researched:** 2026-04-12
**Confidence:** HIGH

## Executive Summary

The shadow realm is a minimal, dark-themed personal site that acts as a public home base and project catalog. Research confirms this is a well-understood domain with a clear best-practice stack: Next.js 15 App Router with Tailwind CSS v4, deployed natively to Vercel via GitHub push. The emphasis throughout is on deliberately not building things — the right design is the simplest one that satisfies the stated requirements, with all scope creep explicitly rejected before it starts.

The recommended approach is a fully static site (SSG) generated by Next.js App Router and served from Vercel's CDN edge network. No `output: 'export'` flag is needed — Vercel handles native Next.js output automatically, preserving future optionality (API routes, image optimization) at zero cost. The math flash card game integrates cleanly as a standalone HTML/JS project in `public/static/math-flashcards/`, embedded via `<iframe>` on a Next.js project detail page. This is the correct microfrontend boundary and requires no rewriting.

The primary risks are configuration pitfalls with Tailwind v4 (configuration format changed completely from v3 — old tutorials are wrong) and DNS misconfiguration when wiring the custom domain. Both are well-documented and avoidable. No architectural risk exists for this project: the complexity ceiling is very low, all patterns are established, and the research is highly confident across all four areas.

## Key Findings

### Recommended Stack

Next.js 15 with Tailwind v4 is the unambiguous recommendation. Both are mature, Vercel-native, and have official integration guides that exactly describe this use case. The `create-next-app` scaffolding gives a correct starting point with zero manual wiring. The only additional dependency for v1 is `next-themes` — though since this site is permanently dark, even that can be replaced by a hardcoded `className="dark"` on the `<html>` element (the simpler choice).

Framer Motion, shadcn/ui, state managers, CMS, analytics, and auth are all explicitly out of scope. The research unanimously recommends against adding any of them to v1. Plain Tailwind utilities are sufficient for a landing page; a TypeScript array in `lib/projects.ts` is the correct "CMS" for a catalog that will have fewer than 10 entries for the foreseeable future.

**Core technologies:**
- **Next.js 15.x**: Framework — App Router default, Vercel-native, SSG pages served from CDN without `output: 'export'`
- **React 19.x**: UI runtime — bundled with Next.js 15, no separate install or pinning needed
- **TypeScript 5.x**: Type safety — generated by `create-next-app` by default, catches errors before deploy
- **Tailwind CSS 4.x (4.2.2)**: Styling — CSS-first config, no `tailwind.config.js` needed, dark mode via `.dark` class, 5x faster builds than v3
- **next/font (Geist)**: Typography — built into Next.js, zero network requests at runtime, matches minimal dark aesthetic
- **Vercel (Hobby)**: Hosting + CI/CD — GitHub push triggers automatic preview deploys, free tier sufficient
- **Node.js 18.18+**: Build runtime — Next.js 15 minimum, Vercel environment satisfies this automatically

### Expected Features

The feature research strongly validates the project brief: the table stakes list is short, the differentiators are modest, and the anti-features list is long. For v1, the goal is to ship the six items below and nothing else.

**Must have (table stakes):**
- Identity header ("this is the shadow realm", lowercase) — the primary signal to visitors
- Short purpose statement / blurb — prevents visitor confusion and immediate bounce
- Project catalog section with at least one live entry (math flash card game) — a hub without outbound links is a dead end
- Fast load and mobile-readable layout — SSG + Vercel handles load; single-column layout handles mobile
- Consistent dark theme — owner has committed to this publicly; inconsistency breaks the aesthetic contract
- Favicon — a missing favicon is a minor but noticeable quality signal in browser tabs

**Should have (differentiators — v1 if low effort, v2 otherwise):**
- Subtle visual or graphic element — prevents the page feeling like a bare `<h1>`, gives it personality without noise
- Intentional lowercase typography — signals the minimalism is deliberate
- Project card metadata (tech used, status, brief description) — adds credibility to catalog entries
- Custom domain — makes the hub feel permanent and owned

**Defer (v2+):**
- Contact form, about/bio, blog, skill bars, scroll animations, light/dark toggle, analytics, CMS, social feed, search, loading screen — all explicitly out of scope per research and PROJECT.md

### Architecture Approach

The entire architecture is a single Next.js App Router site: one root layout, one landing page, one project catalog index, and dynamic `[slug]` pages for individual project entries. The project data lives in a typed TypeScript array (`lib/projects.ts`) — no CMS, no database, no Contentlayer. Standalone HTML/JS projects (math flash cards) are placed in `public/static/<slug>/` and embedded via `<iframe>` in their corresponding Next.js project pages. All pages are pre-rendered at build time via `generateStaticParams()`; no client-side data fetching occurs. There is no server at runtime — Vercel's CDN serves static HTML from edge nodes.

**Major components:**
1. `app/layout.tsx` — HTML shell, dark class hardcoded, Geist font, global nav (Server Component)
2. `app/page.tsx` — Hero/landing section with identity header, blurb, catalog teaser (Server Component)
3. `app/projects/page.tsx` — Full project catalog grid, reads from `lib/projects.ts` at build time (Server Component)
4. `app/projects/[slug]/page.tsx` — Individual project page with iframe embed (Server Component; Client Component only if iframe resize needed)
5. `app/_components/ProjectCard.tsx` — Single catalog entry card component (Server Component unless hover state needed)
6. `lib/projects.ts` — Typed project metadata array, the only "data layer" this project needs
7. `public/static/<slug>/` — Standalone HTML/JS projects, served verbatim by Vercel CDN, completely isolated from Next.js

### Critical Pitfalls

1. **Tailwind v4 config format mismatch** — v4 dropped `tailwind.config.js` and `@tailwind` directives entirely. Use `@import "tailwindcss"` in `globals.css` and `@tailwindcss/postcss` in `postcss.config.mjs`. Any v3 tutorial will lead you wrong. Verify installed version with `npm ls tailwindcss` before following any tutorial.

2. **Adding `output: 'export'` when deploying to Vercel** — This flag is for non-Vercel static hosts only. On Vercel it disables image optimization, route handlers, and middleware for zero benefit. The default `next build` without the flag produces static HTML served from Vercel's CDN. Do not add it.

3. **Public folder filename collision with a Next.js route** — If `public/flashcards.html` exists and `app/flashcards/` is also a route, the build throws a hard error. Use a subdirectory with no corresponding route: `public/static/math-flashcards/index.html`, not `public/math-flashcards.html`.

4. **Dark mode hydration flash** — If dark class is applied dynamically (via `next-themes` or JS), the server-rendered HTML has no class and a visible flash from light to dark occurs on load. Since this site is permanently dark, the fix is trivial: hardcode `className="dark"` on `<html>` in `layout.tsx`. No dynamic theme logic needed.

5. **DNS misconfiguration on custom domain** — Apex domain (`yourdomain.com`) requires an A record pointing to Vercel's IP (`76.76.21.21`), not a CNAME. Subdomains (`www`) use a CNAME to `cname.vercel-dns.com`. Do not include the full domain name in the "Name" field — registrars append the domain automatically.

## Implications for Roadmap

### Phase 1: Project scaffolding and CI/CD pipeline
**Rationale:** The primary stated goal of v1 is "GitHub to Vercel CI/CD pipeline wired up and verified with a live deploy." Nothing else matters until a working deploy exists. This phase has zero user-visible features but proves the entire deploy chain.
**Delivers:** Initialized Next.js 15 repo committed to GitHub, Vercel project connected, automatic deploys triggering on push, custom domain wired (if desired at this stage)
**Addresses:** GitHub to Vercel deploy pipeline (active requirement)
**Avoids:** Tailwind v4 config mismatch (set up correctly from day one); `output: 'export'` anti-pattern (never added); DNS pitfall (tackled in isolation before content exists)

### Phase 2: Core landing page
**Rationale:** With the deploy pipeline working, the site needs its identity. This phase delivers the design language — dark palette, typography, header, blurb, and subtle visual element — before adding the catalog. Establishing visual design before catalog structure prevents rework.
**Delivers:** Live dark landing page with identity header ("this is the shadow realm"), purpose blurb, subtle graphic element, Geist font, favicon
**Uses:** Tailwind CSS v4 dark theme (hardcoded `className="dark"` on `<html>`), `next/font` in Server Component layout
**Implements:** `app/layout.tsx` root shell + `app/page.tsx` hero section
**Avoids:** Dark mode hydration flash (hardcoded dark class); `next/font` production failure (defined in Server Component layout only)

### Phase 3: Project catalog and math flash card game integration
**Rationale:** Once the site's visual identity is established, the catalog is the core value proposition. This phase adds `lib/projects.ts`, the catalog index page, dynamic `[slug]` pages, and integrates the math flash card game as the first live entry via iframe embed.
**Delivers:** Full project catalog section; math flash card game accessible at `/projects/math-flashcards` via iframe; `public/static/math-flashcards/` hosting the standalone HTML/JS project
**Uses:** `lib/projects.ts` typed array, `generateStaticParams()`, `<iframe>` embed pattern, `public/static/` directory
**Implements:** `app/projects/page.tsx`, `app/projects/[slug]/page.tsx`, `app/_components/ProjectCard.tsx`
**Avoids:** Public folder naming collision (use `public/static/` not `public/projects/`); rewriting the game as React (it ships as-is via iframe)

### Phase Ordering Rationale

- Pipeline first because a broken deploy blocks everything; the sooner it is verified, the sooner the rest of development is low-friction.
- Visual identity before catalog because the design language informs how the catalog renders; reversing this order creates rework when palette/typography decisions are made late.
- Catalog and project integration last because it depends on both a working deploy (Phase 1) and an established layout (Phase 2). The math flash card game integration is a single iframe — it should not take a full phase on its own.
- Three phases is correct for this scope. More phases would be over-engineering; fewer would conflate infrastructure setup with feature delivery.

### Research Flags

Phases with standard, well-documented patterns (skip `/gsd-research-phase`):
- **Phase 1 (scaffolding/CI):** `create-next-app` + Vercel GitHub integration is zero-config and exhaustively documented. No research needed — follow official setup guide.
- **Phase 2 (landing page):** Tailwind v4 dark theme + `next/font` patterns are documented in official guides. No research needed.
- **Phase 3 (catalog + iframe):** The `public/static/` + `<iframe>` pattern is confirmed by official Next.js docs and community discussion. No research needed.

No phases in this project require deeper research. The domain is simple, all patterns are established, and the research files contain specific, actionable implementation details for every decision.

Mitigation: deploy to Vercel without the flag in Phase 1 and verify the deploy log confirms static HTML output from CDN — this closes the gap empirically before any content is built.

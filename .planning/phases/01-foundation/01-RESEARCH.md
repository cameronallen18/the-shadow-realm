## Summary

Phase 1 delivers the entire CI/CD pipeline before any UI work begins. The repo already exists on GitHub (`cameronallen18/the-shadow-realm`) with only `README.md`, `CLAUDE.md`, and `.planning/` committed. The scaffolding step must write Next.js files INTO that existing directory — not create a subdirectory. The Vercel connection is a one-time dashboard action (no CLI required). The phase is complete when a push to `main` triggers a successful Vercel build and a public URL loads without errors.

**Critical discovery:** `npx create-next-app@latest` is `16.2.3` as of April 2026 (Next.js 16, not 15) [VERIFIED: npm registry]. The project research in CLAUDE.md recommended Next.js 15.x for stability, but npm's `latest` tag now resolves to Next.js 16. To get Next.js 15, use `npx create-next-app@next-15-3` or pin with `create-next-app@15.x`. Given the project's static-site nature, either works, but explicit version pinning is required to get 15.x.

**Primary recommendation:** Scaffold using `npx create-next-app@next-15-3 .` inside the repo root after temporarily relocating `.git`, then restore `.git` and push. Connect to Vercel via dashboard import. No additional config needed for Vercel compatibility.

## Architecture Patterns

### Scaffolding Into an Existing Git Repository

This is the single most important procedural finding for Phase 1.

**The problem:** Running `npx create-next-app . ` inside a directory that already contains a `.git` folder fails with `EACCES: permission denied` when the tool tries to write to `.git/objects/`. This is a known issue [VERIFIED: github.com/vercel/next.js/issues/46651].

**The workaround — temporary `.git` relocation:**

```bash
# Step 1: Move .git out of the way
cd /home/cameron/projects/the-shadow-realm
mv .git /tmp/shadow-realm-git

# Step 2: Scaffold (directory has existing files — README.md, CLAUDE.md, .planning/)
# create-next-app will ask about existing files; answer to overwrite only conflicting ones
npx create-next-app@next-15-3 . \
  --typescript \
  --tailwind \
  --eslint \
  --app \
  --no-src-dir \
  --import-alias "@/*" \
  --turbopack

# Step 3: Restore .git
mv /tmp/shadow-realm-git .git

# Step 4: Verify git log is intact
git log --oneline -3
```

**What create-next-app generates with these flags:**
```
the-shadow-realm/
├── app/
│   ├── favicon.ico
│   ├── globals.css        # @import "tailwindcss" (v4 format)
│   ├── layout.tsx
│   └── page.tsx
├── public/
│   └── (SVG assets)
├── .eslintrc.json         # OR eslint.config.mjs in Next 15
├── next.config.ts
├── package.json
├── postcss.config.mjs     # @tailwindcss/postcss plugin
├── tsconfig.json
├── README.md              # create-next-app overwrites this — expected
└── .gitignore             # merges or overwrites existing
```

**Existing files that may conflict:**
- `README.md` — create-next-app will overwrite; acceptable (was empty)
- `.planning/` — not touched by create-next-app (different directory)
- `CLAUDE.md` — not touched (not a Next.js file name)

[ASSUMED: create-next-app does not prompt about non-conflicting existing files; only overwrites files it generates]

## Common Pitfalls

### Pitfall 1: Using `npx create-next-app@latest` (Gets Next.js 16, Not 15)

**What goes wrong:** `npm view next dist-tags.latest` = `16.2.3`. Running `npx create-next-app@latest` scaffolds Next.js 16, not 15. CLAUDE.md specifies 15.x as the intended version.

**Why it happens:** npm `latest` tag was updated to Next.js 16 when it hit stable. The CLAUDE.md research was done when 15.x was latest.

**How to avoid:** Use `npx create-next-app@next-15-3` or `npx create-next-app@15` to explicitly get the 15.x line.

**Note:** Next.js 16 would also work for this project (static site, same patterns). The choice is which to commit to. CLAUDE.md says 15.x — use that unless the user decides to update.

[VERIFIED: npm registry — `next dist-tags.latest = 16.2.3`, `next dist-tags.next-15-3 = 15.3.9`]

### Pitfall 2: Running create-next-app Without .git Workaround

**What goes wrong:** Running `npx create-next-app .` in a directory with `.git` fails with `EACCES: permission denied` on `.git/objects/`.

**How to avoid:** Use the temporary `.git` relocation workaround documented in Architecture Patterns above.

[VERIFIED: github.com/vercel/next.js/issues/46651]

### Pitfall 3: Adding `output: 'export'` to next.config.ts

**What goes wrong:** Disables image optimization, route handlers, and edge middleware. No benefit on Vercel.

**How to avoid:** Leave `next.config.ts` with an empty `nextConfig = {}`. This is the correct config for Vercel-native deployment.

[VERIFIED: project PITFALLS.md, STACK.md]

### Pitfall 4: Selecting `--src-dir` During Scaffold

**What goes wrong:** Creates `src/app/` instead of `app/` at root. Both work, but the project's planned architecture (from SUMMARY.md) uses root-level `app/`. Choosing `--src-dir` means every file path in future planning would need adjustment.

**How to avoid:** Use `--no-src-dir` flag to keep `app/` at the project root.

[ASSUMED: SUMMARY.md architecture diagrams use root-level paths; `--no-src-dir` is the matching config]

### Pitfall 5: Vercel Build Failing on First Deploy Due to TypeScript Errors

**What goes wrong:** `create-next-app` default `page.tsx` has a complex example with inline SVGs and complex JSX. TypeScript strict mode is on. If any modifications to the generated files introduce type errors, the Vercel build fails.

**How to avoid:** For Phase 1, commit the scaffold output EXACTLY as generated — do not edit any files before the first deploy. Verify the build passes on scaffold output only, then make changes in subsequent commits.

[ASSUMED: scaffold output compiles clean; this is based on general experience with create-next-app output]

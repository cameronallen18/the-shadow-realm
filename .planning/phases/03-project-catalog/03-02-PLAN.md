**Step 0 — Analyze the source**

Before writing any React, read the provided HTML/JS source and identify:
- What arithmetic operations are included (addition, subtraction, multiplication, division — which?)
- How questions are generated (random operands? difficulty levels? fixed set?)
- How answers are validated (exact match? tolerance?)
- What score tracking exists (correct count, total count, streak?)
- Any timer or round system

Replicate the exact mechanics from the source — do not invent or simplify. If the source has difficulty levels, keep them. If it has a timer, keep it. The goal is faithful conversion, not a new game.

**Step 1 — Create the route file**

Create `app/projects/math-flashcards/page.tsx`. This is a Next.js App Router page — add `"use client"` directive at the top since the game is interactive (useState, event handlers).

**Step 2 — Component structure**

Organize into three components in the same file (or split to separate files in the same directory if complexity warrants):

```
MathFlashcardsPage (default export)
  └─ GameBoard       — renders current question, input, submit
  └─ ScoreTracker    — tracks and displays score, accepts optional onSave prop
```

**ScoreTracker interface (D-11 — extensibility for future leaderboard):**
```typescript
interface ScoreTrackerProps {
  correct: number;
  total: number;
  onSave?: (correct: number, total: number) => void; // future auth hook
}
```
Do NOT implement `onSave`. Just declare the prop. This satisfies D-11 without building auth.

**Step 3 — Game implementation**

Implement the game mechanics faithfully from the source. Use `useState` for:
- `currentQuestion` — the active question object `{ operand1, operand2, operator, answer }`
- `userInput` — the current answer field value
- `correct` and `total` — score counters (pass these to ScoreTracker)
- `feedback` — 'correct' | 'incorrect' | null — brief feedback state after submission

Generate questions in a `generateQuestion()` function outside the component (pure function, easier to test later).

**Step 4 — Dark theme + kid-legible UI (D-15, D-17)**

The game page inherits `background: #0a0a0a` and `color: #ededed` from layout. Apply these conventions:

- Question display: large, centered, minimum `text-4xl` (kids need it big). Use `text-[#ededed]`.
- Answer input: `text-center text-xl` at minimum. Style with `bg-transparent border border-[#ededed]/30 text-[#ededed] rounded px-4 py-2 focus:outline-none focus:border-[#ededed]/60` — no white box, stays dark.
- Submit button: `bg-[#ededed]/10 hover:bg-[#ededed]/20 text-[#ededed] border border-[#ededed]/20 rounded px-6 py-2 text-sm` — subtle, not loud.
- Feedback: brief text — "correct" in `text-[#c8cdd4]`, "try again" in `text-[#ededed]/50`. No color-coded green/red (no warm tones per D-04 from Phase 2 context).
- Score display: `text-[#ededed]/60 text-sm` — secondary, not the focus.
- `duration-0` on any hover transitions (D-16 — no animations).

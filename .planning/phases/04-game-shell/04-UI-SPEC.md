## Scope Note

Phase 4 delivers two visual surfaces:

1. **Catalog entry** — a new row in the existing project list on the landing page (`/`). Rendered by the same `projects.map()` loop that renders the math flash cards entry. No new visual pattern needed; the entry inherits landing page styles exactly.

## Spacing Scale

Standard 8-point scale. All values are Tailwind utility classes (Tailwind v4, no config file).

| Token | Value | Tailwind Equivalent | Usage |
|-------|-------|---------------------|-------|
| xs | 4px | `gap-1`, `p-1` | Icon gaps, tight inline spacing |
| sm | 8px | `gap-2`, `p-2` | Compact element spacing |
| md | 16px | `gap-4`, `p-4` | Default element spacing, overlay internal padding |
| lg | 24px | `gap-6`, `p-6` | Overlay section separation |
| xl | 32px | `gap-8`, `p-8` | Layout gaps |
| 2xl | 48px | `gap-12`, `p-12` | Major section breaks |
| 3xl | 64px | `gap-16`, `p-16` | Page-level spacing (not used in game page) |

**Exceptions:**
- HUD score display: `top-4 right-4` (16px from viewport edges) — matches the md token.
- Touch target minimum for the "tap to start" / "restart" affordance: 44px tall hit area. Use `py-3` (12px padding top + bottom) on text that is ~20px tall = 44px total. This satisfies iOS minimum touch target.

**Source:** Existing codebase uses `gap-6`, `px-6`, `pb-24`, `p-4` throughout. 8-point scale is already the de facto standard. Confirmed from `app/page.tsx` and `app/projects/math-flashcards/page.tsx`.

## Typography

All type is Geist Sans. Geist Mono is used for numeral/score display only.

| Role | Size | Weight | Line Height | Letter Spacing | Usage |
|------|------|--------|-------------|----------------|-------|
| Label / UI micro | 10px (`text-[10px]`) | 400 regular | 1.5 | `tracking-widest` (~0.1em) | Section labels, uppercase tags |
| Body / caption | 12–14px (`text-xs` / `text-sm`) | 400 regular | 1.5 | `tracking-wide` | Overlay instruction text, catalog description |
| Overlay heading | 14px (`text-sm`) | 400 regular | 1.2 | default | "game over", "samus run" idle label |
| Score / HUD | 12px (`text-xs`) | 400 regular | 1.0 | none | Live score counter during play, score on game over |

**Weights used: exactly 2 — regular (400) and none (no bold in game overlays).**
The game overlays are intentionally low-contrast and subdued — no bold weights. Bold is reserved for primary CTAs in other contexts (e.g. math flash cards START button). Game shell avoids it to maintain the dark/minimal aesthetic.

**Monospace:** Score numerals use `tabular-nums` to prevent layout shift as digits change. Geist Mono is available via `--font-mono` CSS variable; use `font-mono` Tailwind class for the score HUD.

**Source:** Derived from existing codebase patterns. `text-sm tracking-wide`, `text-xs uppercase tracking-widest`, and `tabular-nums` are all present in `app/projects/math-flashcards/page.tsx`. Phase 4 uses the same scale subset.

## Color

Site palette is fully established. Phase 4 uses a strict subset — game overlays are even more minimal than the landing page.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#0a0a0a` | Game container background (fullscreen), placeholder canvas background |
| Secondary (30%) | `#ededed` at `/60` opacity (`#ededed99`) | Overlay text at mid-prominence; "tap to start" label |
| Accent (10%) | `#ededed` at full opacity | Restart button text, "game over" heading |
| Muted | `#9ba3ad` | Score value on game over screen; HUD score during play |
| Destructive | N/A | No destructive actions in this phase |

**60/30/10 breakdown for game page:**
- 60% dominant: the `#0a0a0a` fullscreen background occupies nearly the entire viewport.
- 30% secondary: dimmed text (`#ededed/60`) for labels, instructions, idle state copy.
- 10% accent: full-brightness `#ededed` is reserved for the single most important element on each screen — the primary action label or heading.

**Accent reserved for:**
- Idle screen: the "tap to start" button label (or equivalent CTA text)
- Game over screen: the "game over" heading text
- Game over screen: the "restart" button label

**Do not use warm tones.** No `red`, `orange`, `yellow`, or any warm hex in overlays. All overlays are cool-tone gray scale only.

**Catalog entry** on landing page inherits the existing landing page color contract without changes:
- Name: `text-[#ededed]`
- Description: `text-[#ededed]/40`

**Source:** CLAUDE.md constraint (cool tones only). STATE.md confirmed Phase 3 timer ring was adapted from warm colors to `#9ba3ad`/`#c8cdd4`. Existing globals.css defines `#0a0a0a` background and `#ededed` foreground. Research anti-patterns section explicitly names the three approved hex values.

## Copywriting Contract

All copy is lowercase, minimal, understated — consistent with the site voice established in Phase 2 ("my digital junk drawer.").

| Element | Copy |
|---------|------|
| Catalog entry name | `samus run` |
| Catalog entry description | `norfair escape. dodge the rock walls.` |
| Idle overlay label | `samus run` |
| Idle CTA (primary) | `tap to start` |
| Playing HUD | *(score integer only — no label)* |
| Game over heading | `game over` |
| Game over score label | *(score integer only — no label)* |
| Game over CTA (primary) | `restart` |
| High score label (Phase 4 placeholder) | `best: 0` *(hardcoded 0 until Phase 7 adds localStorage)* |

**Notes:**
- No punctuation on CTAs. Not "tap to start." — just `tap to start`.
- Score display is a bare integer — no "score:" prefix. The context makes the number self-evident.
- "best: 0" on the idle/game-over screens uses the lowercase-colon-value pattern common in the existing codebase (e.g. "Correct", "Avg Speed" labels use all-caps — but this site voice prefers lowercase, so `best: 0` not `BEST: 0`).
- No empty state needed — the game page always has content (the idle overlay is the "no game yet" state).
- No error state UI needed in Phase 4 — no network calls, no user input validation.
- No destructive actions — restart is non-destructive (score resets, no data is permanently deleted in this phase).

**Source:** Catalog copy from 04-RESEARCH.md (Pattern 4, code example). CTA and overlay copy from 04-RESEARCH.md (Pattern 3, JSX example). Voice from STATE.md Phase 2 decisions ("my digital junk drawer." as blurb) and CLAUDE.md ("understated in tone").

## Layout Contract

### Catalog Entry (`lib/projects.ts` + `app/page.tsx`)

No new layout needed. The samus-run entry is appended to the `projects` array. The existing `projects.map()` loop renders it identically to the math flash cards entry. No new CSS, no new components.

### Game Page (`/projects/samus-run`)

**Container:** `w-full h-dvh` — use `h-dvh` (dynamic viewport height), not `h-screen` or `100vh`. iOS Safari's `100vh` overflows past the browser chrome. `h-dvh` is Tailwind v4 built-in and correct.

**Overflow:** `overflow-hidden` on the root container — prevents any scroll on the game page.

**Position model:** `relative` on root container. All overlays are `absolute inset-0` children.

**Overlay alignment:**
- Idle: `flex flex-col items-center justify-center` — centered vertically and horizontally.
- Playing HUD: `absolute top-4 right-4` — top-right corner, score only.
- Game over: `flex flex-col items-center justify-center` — centered, same as idle.

**Back navigation:** A back link to `/` must appear on the game page. Use the same pattern as math-flashcards: `fixed top-4 left-4 text-[#ededed]/40 text-xs`. Copy: `← the shadow realm`. This overlays the game container without disrupting layout.

**Source:** 04-RESEARCH.md Pitfall 2 (h-dvh requirement). Pattern 3 layout code. Math-flashcards back link pattern from `app/projects/math-flashcards/page.tsx` line 485–490.

## Interaction Contract

### State Machine Transitions

| From | Action | To | Visual Change |
|------|--------|----|---------------|
| `idle` | Click / tap on "tap to start" | `playing` | Idle overlay unmounts; HUD score `0` appears top-right |
| `playing` | `dispatch({ type: "GAME_OVER", score: N })` called from code | `gameover` | HUD unmounts; game over overlay mounts with final score |
| `gameover` | Click / tap "restart" | `idle` | Game over overlay unmounts; idle overlay mounts; score resets to 0 |

### Interactive Elements

| Element | State | Interaction | Visual Response |
|---------|-------|-------------|-----------------|
| "tap to start" label/button | idle | click / tap | Transitions to `playing` state |
| "restart" label/button | gameover | click / tap | Transitions to `idle` state |
| Back link "← the shadow realm" | all | click | Navigates to `/` (Next.js Link) |

### No Hover States on Game Controls

The game is touch-first. Hover effects are intentionally omitted on "tap to start" and "restart" to avoid visual noise. The back link may use `hover:text-[#ededed]/70` to match the math-flashcards pattern — this is a nav element, not a game control.

### Transition Duration

**None.** All state transitions are instant (`duration-0` or no transition class). The site design aesthetic explicitly avoids animation. Overlay swaps are immediate.

**Source:** CLAUDE.md: "animations are counter to the vibe." Existing codebase uses `transition-colors duration-0` throughout math-flashcards to neutralize Tailwind's default transitions.
